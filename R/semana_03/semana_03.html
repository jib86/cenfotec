Tema 1

¿Qué es el Análisis Exploratorio?

¿Qué es?

EDA (Exploratory Data Analysis) es el proceso de conocer tus datos antes de modelarlos. Lo inventó el estadístico John Tukey en los años 60. La idea: mirar primero, modelar después.

La idea en simple:
Si saltás el EDA y construís el modelo de inmediato, es como cocinar sin probar los ingredientes. El 80% de los errores en modelos de datos vienen de no haber explorado primero.

¿Qué buscamos?

  • Forma de la distribución de cada variable
  • Outliers — valores extremos o errores
  • Nulos — datos faltantes y su patrón
  • Relaciones entre variables

Flujo de trabajo

  1. Dimensión y tipos de datos
  2. Estadísticas resumen
  3. Visualizar distribuciones
  4. Identificar correlaciones
  5. Reportar hallazgos

Mentalidad EDA

¿Qué gráfico para qué?

Histograma: ¿Es simétrica mi distribución, o sesgada? ¿Hay dos picos (bimodal)?

Scatter plot: Detecta relaciones lineales (nube diagonal = correlación). El parámetro color = Species añade una dimensión extra sin agregar ejes.

geom_*() → el tipo de gráfico
labs() → las etiquetas
theme_*() → el estilo visual

#               Sepal.L Sepal.W Petal.L Petal.W
# Sepal.Length     1.00   -0.12    0.87    0.82
# Sepal.Width     -0.12    1.00   -0.43   -0.37
# Petal.Length     0.87   -0.43    1.00    0.96
# Petal.Width      0.82   -0.37    0.96    1.00

# Visualizar la matriz como mapa de calor
corrplot(matriz_cor,
         method  = "color",   # Usa colores sólidos
         type    = "upper",   # Solo triángulo superior (evita repetición)
         addCoef.col = "black",  # Pone el número dentro de cada celda
         tl.col  = "white",   # Color de las etiquetas
         col     = colorRampPalette(c("#00e5ff","white","#9d4edd"))(200))

¿Qué significa r?

El coeficiente de correlación de Pearson (r) mide qué tan fuerte y en qué dirección se mueven juntas dos variables numéricas:

r = +0.87 → Correlación positiva fuerte. Si una sube, la otra sube.
r = -0.43 → Correlación negativa moderada. Si una sube, la otra baja.
r ≈ 0.00 → Sin relación lineal. No implica que no haya relación de otro tipo.
¡Correlación ≠ Causalidad!

Las ventas de helado correlacionan con los ahogamientos. Ambos los causa el verano, no uno al otro.

La idea en simple:
Te dice cuáles variables valen la pena meter en el modelo y cuáles son redundantes. Si dos variables tienen r = 0.98, es casi lo mismo meterlas ambas — con una alcanza, y el modelo queda más limpio.

Tema 3

Manejo de Nulos (NA)

manejo_nulos.R 
# Creamos datos CON nulos para practicar
set.seed(42)  # Semilla: hace que lo "aleatorio" sea reproducible
datos_na <- airquality  # Dataset de calidad del aire (incluye NAs reales)

# --- DETECTAR ---
sum(is.na(datos_na))           # Total de NAs en todo el dataset: 44
colSums(is.na(datos_na))       # NAs por columna
# Ozone  Solar.R Wind Temp Month Day 
#    37        7    0    0     0   0

# ¿Qué porcentaje? (regla: > 5% requiere decisión cuidadosa)
round(colMeans(is.na(datos_na)) * 100, 1)  # Ozone: 24.2% ⚠️

# --- ESTRATEGIA 1: Eliminar filas con NA (na.omit) ---
datos_limpios <- na.omit(datos_na)
nrow(datos_limpios)  # De 153 quedamos en 111 filas

# --- ESTRATEGIA 2: Imputar con la mediana (más robusta que la media) ---
datos_imputados <- datos_na |>
  mutate(Ozone = ifelse(is.na(Ozone), median(Ozone, na.rm = TRUE), Ozone),
         Solar.R = ifelse(is.na(Solar.R), median(Solar.R, na.rm = TRUE), Solar.R))

sum(is.na(datos_imputados))  # 0 — ¡sin NAs!

¿Eliminar o imputar?

Eliminar (na.omit): Opción segura si el % de NAs es bajo (<5%) y los NAs son aleatorios (no hay patrón). Riesgo: pierdes filas valiosas.

Imputar con mediana: Preferible a la media cuando hay outliers. La mediana no se "distorsiona" por valores extremos.

Imputar con media: Funciona bien en distribuciones simétricas sin outliers.

¿Por qué na.rm = TRUE? Si no ponés este argumento, mean() y median() devuelven NA si hay aunque sea un solo NA en el vector. Este parámetro le dice: "ignorá los NAs al calcular".

Error común:

mean(Ozone) → devuelve NA si hay NAs.
mean(Ozone, na.rm = TRUE) → devuelve el número correcto.

La idea en simple:
La mayoría de funciones de R simplemente fallan o dan resultados silenciosamente incorrectos si hay NAs. Tratar los nulos no es opcional — es el pasaporte para que cualquier análisis funcione.

Tema 4

Resampling: Train / Test Split

resampling.R 
datos <- na.omit(airquality)  # Partimos de datos limpios

# set.seed() es CRÍTICO: garantiza resultados reproducibles.
# Cualquier número funciona; el mismo número siempre da el mismo resultado.
set.seed(2024)

# ¿Cuántas filas tenemos?
n <- nrow(datos)   # 111 filas

# Generar índices aleatorios para el 80% de las filas (entrenamiento)
indices_train <- sample(1:n, size = round(0.80 * n))

# Dividir el dataset
train <- datos[ indices_train, ]   # 80% para entrenar el modelo
test  <- datos[-indices_train, ]   # 20% para evaluarlo (el "-" excluye)

# Verificar la división
nrow(train)   # 88 filas
nrow(test)    # 23 filas

# ¿Mantienen proporciones similares? Verificar con summary()
cat("Train - Media Ozone:", round(mean(train$Ozone), 1), "\n")
cat("Test  - Media Ozone:", round(mean(test$Ozone),  1), "\n")

¿Por qué dividir?

Si entrenas y evalúas con los mismos datos, tu modelo siempre se ve perfecto. Es como estudiar el examen con las respuestas en mano.

Train (80%): El modelo aprende los patrones con estos datos.

Test (20%): Datos que el modelo nunca vio. Aquí medimos si realmente aprendió, o si solo memorizó.

set.seed(): Congela el generador de números aleatorios. Sin él, cada vez que corras el código, la división será distinta y los resultados cambiarán. Con él, cualquier persona que corra tu código obtendrá exactamente el mismo split.

Regla 80/20:

Es la más común. Con datasets pequeños (< 200 filas) podés usar 70/30. Para datasets muy grandes, hasta 95/5 es razonable.

La idea en simple:
Sin esta división, tu modelo podría tener un R² de 0.99 y aun así ser inútil en el mundo real. El test set es la única prueba honesta de que tu modelo aprendió algo verdadero, no que memorizó los datos.

Tema 5

Regresión Lineal

regresion_lineal.R 
datos <- na.omit(airquality)
set.seed(2024)
n <- nrow(datos)
train <- datos[sample(1:n, round(0.8*n)), ]
test  <- datos[-sample(1:n, round(0.8*n)), ]

# AJUSTAR MODELO: ¿puede el viento y la temperatura predecir el ozono?
# Fórmula: Ozone ~ Wind + Temp  →  "Ozone EN FUNCIÓN DE Wind y Temp"
modelo_lm <- lm(Ozone ~ Wind + Temp, data = train)

# EXPLORAR EL MODELO en detalle
summary(modelo_lm)
# Coefficients:
#             Estimate Std. Error t value Pr(>|t|)    
# (Intercept) -71.033     23.470  -3.026  0.00333 ** 
# Wind         -3.045      0.664  -4.586  1.8e-05 ***
# Temp          1.839      0.254   7.237  5.8e-10 ***
# 
# R-squared: 0.5687  → el modelo explica el 56.9% de la varianza del Ozono

# PREDECIR sobre datos nuevos (test)
pred_lm <- predict(modelo_lm, newdata = test)

# CALCULAR ERROR: RMSE (raíz del error cuadrático medio)
rmse <- sqrt(mean((test$Ozone - pred_lm)^2))
cat("RMSE:", round(rmse, 2), "ppb de ozono\n")

Leer el summary()

Estimate (coeficiente): Efecto de cada variable. Wind = -3.04 significa que por cada milla/hora más de viento, el ozono baja ~3 ppb, manteniendo la temperatura constante.

Pr(>|t|) — p-value: Si es < 0.05, la variable es estadísticamente significativa. Los *** indican p < 0.001 (muy significativo).

R² (R-squared): Proporción de varianza explicada. Va de 0 a 1. Un R² = 0.57 en ciencias ambientales es razonablemente bueno.

RMSE: El error promedio en las mismas unidades que la variable respuesta. Más intuitivo que R².

La idea en simple:
Usala cuando querés predecir un número. ¿Cuánto costará esta casa? ¿Qué temperatura habrá mañana? ¿Cuántas ventas habrá en diciembre? Si la respuesta es un valor continuo, empezá por aquí.

Tema 5 — Diagnóstico

¿Mi modelo es válido? — Residuos

diagnostico_residuos.R 
datos  <- na.omit(airquality)
modelo_lm <- lm(Ozone ~ Wind + Temp, data = datos)

# R genera 4 gráficos de diagnóstico automáticamente con una sola línea
# Ponemos par(mfrow) para verlos todos juntos en una grilla 2x2
par(mfrow = c(2, 2))
plot(modelo_lm)
par(mfrow = c(1, 1))  # Restaurar la pantalla a normal

# ── Los 4 gráficos que aparecen ──────────────────────────────
# 1. Residuals vs Fitted:  ¿Los errores son aleatorios?
#    ✅ Bueno: puntos dispersos sin patrón
#    ❌ Malo:  curva o embudo → el modelo no captura algo

# 2. Normal Q-Q:  ¿Los residuos siguen una distribución normal?
#    ✅ Bueno: puntos sobre la línea diagonal
#    ❌ Malo:  colas muy desviadas → los supuestos se rompen

# 3. Scale-Location:  ¿La varianza es constante? (homocedasticidad)
#    ✅ Bueno: línea roja aproximadamente horizontal
#    ❌ Malo:  línea con pendiente → heterocedasticidad

# 4. Residuals vs Leverage:  ¿Hay puntos influyentes que distorsionan el modelo?
#    ✅ Bueno: ningún punto cruzando la línea de Cook's distance
#    ❌ Malo:  un punto en la esquina superior derecha = outlier influyente

¿Por qué revisarlos?

Un R² alto no garantiza que el modelo sea correcto. Los supuestos de la regresión lineal exigen que los errores (residuos) sean:

Aleatorios — sin patrón sistemático Normales — distribuidos en campana Homocedásticos — varianza constante

Regla práctica:

Si el gráfico Residuals vs Fitted muestra un patrón curvo, tu modelo está mal especificado. Probá agregar un término cuadrático: lm(y ~ x + I(x^2), data)

Outlier influyente detectado:

Si un punto cruza Cook's distance, investigalo antes de borrarlo. Puede ser un error de carga de datos — o el caso más interesante de todo tu dataset.

La idea en simple:
Un R² de 0.95 no significa que el modelo sea correcto — puede ser que estés violando un supuesto matemático sin saberlo. Los gráficos de residuos son la radiografía que confirma que el modelo está sano por dentro, no solo bien en el papel.

Tema 6

Regresión Logística

regresion_logistica.R 
# Problema: predecir si un auto es Manual (1) o Automático (0)
datos <- mtcars

# La variable respuesta DEBE ser 0 o 1
# 'am': 0 = automático, 1 = manual (ya está en formato correcto)

set.seed(2024)
n <- nrow(datos)
idx <- sample(1:n, round(0.75 * n))
train <- datos[idx,  ]
test  <- datos[-idx, ]

# AJUSTAR MODELO LOGÍSTICO
# family = binomial() le dice a R: "la respuesta es 0/1, usa sigmoide"
modelo_log <- glm(am ~ hp + wt,
                  data   = train,
                  family = binomial())

summary(modelo_log)
# wt (peso) es altamente significativo (***): autos más pesados → más automáticos

# PREDECIR probabilidades sobre test
prob_manual <- predict(modelo_log, newdata = test, type = "response")
# type = "response" convierte log-odds a probabilidades (0 a 1)

# CLASIFICAR con umbral 0.5
pred_clase <- ifelse(prob_manual >= 0.5, 1, 0)

# MATRIZ DE CONFUSIÓN
table(Real = test$am, Predicho = pred_clase)
#       Predicho
# Real   0 1
#      0 4 1
#      1 0 3

Lineal vs Logística

Regresión Lineal: La variable respuesta es continua (temperatura, precio, ozono). Predice un número.

Regresión Logística: La variable respuesta es binaria (0/1, sí/no, enfermo/sano). Predice una probabilidad.

La función sigmoide ($\sigma$) transforma cualquier número en un valor entre 0 y 1. Por eso usamos family = binomial().

Matriz de confusión: La tabla que compara real vs predicho. De la diagonal (correcto) y fuera de diagonal (errores) extraemos precisión, recall y F1.

Umbral:

El umbral 0.5 no siempre es el óptimo. En diagnóstico médico, preferirías bajar el umbral a 0.3 para no perder ningún caso positivo (aunque haya más falsos positivos).

La idea en simple:
Usala cuando la respuesta es sí o no. ¿Este correo es spam? ¿Este cliente va a cancelar? ¿Este tumor es maligno? El modelo te devuelve una probabilidad — vos decidís a partir de cuál valor actuás.

¿Por qué se llama sigmoide?

La función sigmoide tiene forma de "S" (la letra griega sigma: σ). Convierte cualquier número real en una probabilidad entre 0 y 1:

σ(x) = 1 / (1 + e⁻ˣ)

add_hline(y = 0.5): Agrega una línea horizontal que indica el umbral de clasificación. Los puntos por encima → clasificados como "Manual".

text = ~paste(...): Personaliza el tooltip que aparece al hacer hover sobre cada punto.

¿Para qué la quiero yo?
Sin la sigmoide, la regresión logística podría predecir "probabilidades" de 150% o -30%, que no tienen ningún sentido. La sigmoide es el candado que fuerza cualquier salida a vivir entre 0 y 1 — donde las probabilidades realmente existen.

Tema 7

Selección de Variables y Diagnóstico

seleccion_variables.R 
library(car)

# --- MODELO COMPLETO (todas las variables) ---
modelo_completo <- lm(mpg ~ ., data = mtcars)
# El punto "." significa: "todas las demás columnas como predictores"

# --- AIC: Akaike Information Criterion ---
# Penaliza la complejidad. Menor AIC = mejor modelo.
AIC(modelo_completo)   # 163.7

# --- step(): Selección automática por AIC (stepwise backward) ---
# Prueba eliminar cada variable de a una. Si al eliminarla el AIC baja → se va.
modelo_opt <- step(modelo_completo, direction = "backward", trace = 0)
# trace = 0 silencia el proceso (trace = 1 lo muestra paso a paso)

summary(modelo_opt)
# Variables seleccionadas: wt, qsec, am  →  R² = 0.8497

AIC(modelo_opt)   # 156.0 → ¡Mejor que el completo con menos variables!

# --- VIF: Variance Inflation Factor (detecta multicolinealidad) ---
# Regla de oro: VIF > 10 = problema. VIF > 5 = sospecha.
vif(modelo_opt)
#        wt      qsec        am
#  2.482952  1.364339  2.541农

# Resultado limpio: ninguna variable está inflada. ¡Buenas noticias!

Los 3 diagnósticos clave

AIC (Akaike): Equilibra bondad de ajuste vs complejidad. Un modelo con más variables no es necesariamente mejor — puede estar "memorizando ruido" (overfitting). Siempre compará modelos con AIC.

step() — Stepwise: Busca automáticamente la mejor combinación de variables. No es perfecto, pero es un excelente punto de partida.

VIF (multicolinealidad): Si dos predictores están muy correlacionados entre sí (ej: hp y disp en mtcars), el modelo se "confunde" y los coeficientes se vuelven inestables. VIF lo detecta. Si VIF > 10, eliminá una de las dos variables correlacionadas.

La idea en simple:
Menos variables no significa peor modelo — a menudo significa mejor. Un modelo con 3 variables claras es más fácil de explicarle a tu jefe, más rápido de calcular y, en la mayoría de casos, comete menos errores con datos nuevos.

Tema 8 — Concepto

¿Qué son Ridge y Lasso?

¿Son apellidos?

No exactamente. Son métodos estadísticos, pero sus nombres sí tienen historia:

Ridge (en inglés: "cresta") fue propuesto por Hoerl y Kennard en 1970. El nombre hace referencia a la forma geométrica de la solución.

Lasso es un acrónimo: "Least Absolute Shrinkage and Selection Operator". Lo propuso Robert Tibshirani en 1996 en Stanford.

El problema: Overfitting

Imaginá que estudiás memorizando las respuestas exactas — pero en el examen real las preguntas son distintas y fallás.

Eso es overfitting: el modelo aprende el ruido de los datos de entrenamiento y falla en datos nuevos.

Ridge y Lasso buscan el modelo que más generaliza, no el que más ajusta.

La diferencia clave

Ridge (L2) — El "dial suavizador"
Achica coeficientes hacia cero, pero nunca los elimina.
Lasso (L1) — El "bisturí"
Lleva coeficientes a exactamente cero. Elimina variables automáticamente.
¿Cuándo usar cuál?
Variables irrelevantes → Lasso. Variables correlacionadas → Ridge.

¿Para qué los uso en la práctica?

Tenés 50+ variables y no sabés cuáles importan
Usá Lasso. Descarta automáticamente las irrelevantes. Ahorra horas de prueba y error manual.
Tus variables están muy correlacionadas entre sí
Usá Ridge. Las suaviza sin eliminar ninguna, manejando mejor la multicolinealidad.
Tu modelo entrena bien pero falla con datos nuevos
Síntoma clásico de overfitting. Cualquiera de los dos disciplina al modelo para que aprenda, no memorice.
Necesitás explicar el modelo a alguien no técnico
Usá Lasso para quedarte con 4-5 variables clave, luego corrés un lm() simple con esas.
Tema 8 — Código

Ridge y Lasso en R con glmnet

regularizacion.R 
library(glmnet)

# glmnet NO acepta data.frames directamente — necesita matrices
# model.matrix() convierte automáticamente variables categóricas a dummies
X <- model.matrix(mpg ~ ., data = mtcars)[, -1]  # [,-1] elimina el intercepto
y <- mtcars$mpg

# RIDGE (alpha = 0): achica coeficientes pero nunca los elimina
modelo_ridge <- glmnet(X, y, alpha = 0)

# LASSO (alpha = 1): puede poner coeficientes exactamente en CERO
modelo_lasso <- glmnet(X, y, alpha = 1)

# Buscar el lambda óptimo con validación cruzada (10 particiones)
cv_lasso <- cv.glmnet(X, y, alpha = 1, nfolds = 10)

# Ver los coeficientes con el lambda que minimiza el error
coefs <- coef(cv_lasso, s = "lambda.min")
print(coefs)

# ── Cómo interpretar la salida ──────────────────────────────
# (Intercept)  37.2851
# cyl           .        ← PUNTO = coeficiente = 0 → Lasso la eliminó
# disp          .        ← esta variable NO aporta al modelo
# hp           -0.0215   ← esta SÍ aporta (coeficiente distinto de 0)
# drat          .
# wt           -3.7152   ← variable más importante (mayor magnitud)
# qsec          .
# vs            .
# am            2.5202
# gear          .
# carb         -0.0785

# Las variables con punto (.) son irrelevantes según Lasso
# Podés usarlas para simplificar tu modelo final
vars_relevantes <- rownames(coefs)[which(coefs != 0)]
cat("Variables seleccionadas:", vars_relevantes, "\n")

El dial λ (lambda)

Lambda controla qué tan fuerte es la penalización:

λ = 0 → regresión lineal normal, sin penalización
λ óptimo → encontrado por cv.glmnet() automáticamente
λ = ∞ → todos los coeficientes → 0, modelo inútil
El punto (.) en la salida:

Cuando Lasso pone un coeficiente en 0, R lo muestra como . para ahorrar espacio. Esa variable no aporta nada al modelo según Lasso — podés excluirla en tu versión final con lm().

Flujo recomendado:

Usá Lasso para descubrir qué variables importan → luego ajustá un lm() clásico solo con esas → así tenés un modelo interpretable y liviano.
